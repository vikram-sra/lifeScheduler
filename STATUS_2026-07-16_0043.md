## 1. Architecture refactor — MVP4 (data.js, render.js)

The core problem going in: the DOM *was* the database. `index.html` hardcoded every cell; `saveSchedule()`/`loadSchedule()` scraped `textContent` in and out of localStorage, with ~180 lines of `if (text.includes('dinner'))`-style guessing to recover meaning on every load. There was no stable slot identity, no category taxonomy, and no way to distinguish "core week" from "this week only" changes.

**New files:**
- **`data.js`** — the model layer, no DOM access. Defines:
  - `CATEGORIES` (work/project/habit/health/meal/rest/commute/admin/other/flow) with colors shared with the CSS token system
  - `BUILTIN_ACTIVITIES` registry (id, label, icon, category, style — identity separated from visual class)
  - `Slot` shape `{ id, day, time, activityId, label, url, fixed }` with a stable id
  - `week = { template, overrides, done, activities }` — template is the recurring core structure; overrides are this-week-only patches keyed by week
  - `resolveWeek(date)`, `createSlot/updateSlot/deleteSlot/moveSlot` (scope-aware: `'template'` vs `'week'`)
  - `migrateFromV3()` — the one place the old text-guessing logic runs, once, converting the legacy `lifeScheduler` localStorage key into the new `day_v4` model (legacy key kept as rollback backup)
  - `seedDefaultModel()` — recreates the original hardcoded week as data
  - `weekRollup()`, `streakFor()` for analytics (streaks since removed, see §5)
- **`render.js`** — `renderWeek()` projects the model into `<tbody>`; the HTML `<tbody>` is now empty and fully data-driven. Stamps `data-slot-id`/`data-activity-id`/`data-category`/`data-fixed` onto cells.

**script.js rewiring:**
- Deleted `saveSchedule()`/`loadSchedule()` (~180 lines of scrape/guess logic) — replaced by `applySlotEdit()` operating directly on the model
- Activity picker rewritten to read/write the registry, with a scope toggle (🔁 Every week / 📅 Just this week)
- Drag-and-drop now moves slots in the model (`moveSlot`) rather than swapping `innerHTML`
- Fixed vs. flex distinction added: fixed slots (meals, sleep) resist drag; flex slots (work, projects, health) can move. Toggle available per-slot in the picker.
- CSV/JSON export, add-time-slot, and reset all rebuilt against the model
- Performance: `highlightCurrentTime()` now gates its per-cell class sweep behind a change-detection key so it only touches the DOM when the active row/day actually changes, not every second

**Verification:** a standalone Node test harness (36 assertions) exercised migration, template vs. week-scoped overrides, week-only adds/deletes, streak math, and persistence round-tripping — all passed before touching the DOM layer.

## 2. UI overhaul — "Refined Midnight"

Full visual rebuild on top of the same layout/structure, per user-selected direction (dark/glass foundation, restrained single-accent glow, softer glass, tighter type, more whitespace; emoji icons kept).

- Introduced a CSS custom-property token system (`--bg-*`, `--accent`, `--gold`, `--cat-*`, `--radius-*`, `--shadow-*`) — previously every color was a hardcoded hex scattered through ~2000 lines
- Category colors in `data.js` now mirror the CSS `--cat-*` tokens, so analytics and any category-driven UI stay visually consistent with the theme by construction
- Removed the alternating red/teal/gold weekday background tints (visual noise); weekdays are now nearly flat, weekend barely warmer, so activity color carries the signal
- Unified the "today / active now" treatment onto a single gold accent (previously 3 different accent colors competed)
- Existing animated SVG activity backgrounds (work, gym, meal, duar, meditate, water, hike, etc.) were left untouched — only their surrounding border/glow/shadow was retoned
- Settings panel, activity picker, and analytics overlay reskinned onto the same token system

## 4. Behavioral fixes (edit mode, scroll, flaps)

- **Exiting edit mode** now re-enables Focus Mode and smooth-scrolls back to today's column (previously left the view wherever the user last scrolled)
- **Past-time fading** (cells dim to 25% opacity once their time has passed) is now suspended while editing or with Focus Mode off — previously past cells stayed hard to see/tap even mid-edit
- **Scroll-jump bug:** `highlightToday()` had an unconditional `setTimeout(scrollToToday, 500)` that reran on *every* call — since `refreshUI()` (fired after any edit) calls `highlightToday()`, tapping a ritual cell on a narrow viewport would auto-scroll the whole table back to today 500ms later, out from under the user. Removed the redundant unconditional call; the legitimate first-load scroll and deliberate re-centers (edit-mode exit, window resize) are untouched.
- **Sleep shutters bug:** the tap-and-hold-to-peek `closeAll()` handler fired on every `pointerup` on the table with no edit-mode guard, so tapping cells while editing during sleep hours could snap the shutters back shut mid-edit. Fixed.

## 5. Habit streaks — added, then removed

Phase 4 of the original MVP4 plan added a done-toggle + 🔥 streak badge on habit/health cells. Two bugs surfaced in testing:
1. The streak badge was computed per-*activity* and broadcast to every occurrence of that activity that day (e.g. all 6 "Water" ritual slots lit up from checking just one)
2. `td.slot-done { position: relative }` collided with the rituals column's `position: sticky`, causing cells to visually detach/float out of their column on scroll

The badge-broadcast bug was fixed first, but the user found the feature "very weird" overall and asked for removal rather than further iteration. **Fully removed**: no more done-toggle on tap, no badges, no analytics Streaks section. The underlying `model.done` data structure and `streakFor()`/`toggleDone()`/`isDone()` functions remain in `data.js` (dead code, harmless) in case a cleaner surface for habit tracking is wanted later — a dedicated checklist strip rather than overlays fighting the cell art.

## 8. Power / performance audit

User reported the app "consuming too much power." Audit found:
1. Every table cell (100+) was forced onto its own GPU compositor layer permanently via a blanket `will-change: transform, opacity` + `translateZ(0)` on the base `td` rule, regardless of whether that cell ever animated
2. Paint/GYM activity glow animations ran as infinite `box-shadow` keyframes (expensive — forces repaint, not just composite) on *every* occurrence of that activity across the whole week — a typical week has ~14 Paint cells and ~7 GYM cells, all animating forever, even on days dimmed to 15% opacity under Focus Mode
3. A 1-second `setInterval` polled continuously, including while the tab/PWA was backgrounded
4. Heavy `backdrop-filter: blur(28px)` on the always-visible dock

**Fixes:**
- Removed the blanket layer promotion from the base `td` rule
- Scoped glow animations to only `.today`/`.active-slot`/`.current-active-cell` — verified only 1 of 14 Paint cells and 1 of 7 GYM cells animate now
- Added a `prefers-reduced-motion` media query killing all decorative infinite animations for users with that OS preference
- Timer: 1000ms → 4000ms polling, plus a `visibilitychange` listener that fully stops the timer when hidden and resyncs immediately on return
- Trimmed blur radii (dock 28→16px, panels 24→18px)

## 9. Cache-busting infrastructure

This session repeatedly hit a caching layer (independent of the browser and the service worker, tied to the preview proxy) that served stale JS/CSS across server restarts and even fresh tabs. Root-caused via direct `curl` against the server and cache-busted `fetch()` calls. Fixed the underlying real-world version of this problem (relevant for actual users updating the PWA, not just this session's tooling):
- `index.html` now references assets with a version query string (`script.js?v=9`, etc.)
- `sw.js` fetch handler uses `{ ignoreSearch: true }` so the cache still matches regardless of the query, while `CACHE_NAME` bumps (now `life-scheduler-v9`) force a full cache purge on activate
